# Hermes Desktop v1.0 Product Spec

Date: 2026-04-13

Status: Draft

## 1. 产品定义

`Hermes Desktop` 是一款基于 Hermes 架构的本地 coding agent 桌面客户端。

它不是对 CLI 的简单壳化，而是把 Hermes 现有能力重新组织成桌面产品：

- 本地运行
- 安装即用
- 一次首启配置
- 面向工作区与代码任务
- 可视化工具执行过程
- 真实审批与澄清交互

一句话定义：

> 一个基于 Electron 的本地 Hermes agent 工作台，让用户安装后即可在 Windows 和 macOS 上使用 Hermes 的 agent 能力，而不需要手动安装任何依赖。

## 2. 问题陈述

当前 Hermes 的核心能力很强，但默认使用门槛仍然偏开发者：

- CLI 安装与环境准备门槛高
- Web UI 现阶段偏 dashboard，不是完整聊天产品
- API server 更像开放接口，不是终端用户产品
- 浏览器工具、Python 依赖、Node 依赖对普通用户不够友好

这意味着 Hermes 已经“有内核”，但还没有“开箱即用的桌面产品层”。

## 3. 产品目标

### 3.1 v1.0 目标

1. 用户下载安装后即可启动 Hermes Desktop。
2. 用户只需在首启时完成一次 provider/API key 配置。
3. 用户无需自己安装 Python、Node、Chromium 或其他运行时。
4. 用户能在桌面客户端里：
   - 发起新对话
   - 选择工作区
   - 看见 agent 的实时输出
   - 看见工具执行时间线
   - 处理审批与澄清
   - 查看历史会话
   - 查看日志、技能、配置
5. 应用在 `Windows` 与 `macOS` 上可稳定运行。

### 3.2 成功标准

- 首次安装到进入可聊天状态，目标不超过 3 分钟
- 用户不需要额外打开终端安装依赖
- 首启设置流程在 4 步内完成
- 常见 coding 会话可以稳定完成完整 agent 回合
- 会话关闭后可恢复历史与上下文

## 4. 目标用户

### Persona A: 本地开发者

特征：

- 平时写代码
- 希望得到比普通聊天更强的本地工具能力
- 不想维护复杂 agent 环境

主要诉求：

- 能直接指定工作区
- 能看清 agent 做了什么
- 能控制危险命令是否执行

### Persona B: 技术 PM / 研究型用户

特征：

- 会做一定技术工作，但不想全程在终端里操作
- 更在意可视化与可解释性

主要诉求：

- 有清晰的会话管理与日志查看
- 能方便切换 provider、模型、技能
- 能看到 agent 当前卡在哪一步

## 5. 核心使用场景

### 场景 1: 首次安装与配置

用户下载 Hermes Desktop 安装包并安装后：

1. 启动应用
2. 进入 Setup Wizard
3. 选择 provider
4. 填写 API key
5. 选择默认模型
6. 选择默认工作区与审批模式
7. 完成后进入主聊天界面

### 场景 2: 新建 coding 会话

用户点击 `New Chat`：

1. 选择或确认工作区
2. 输入任务
3. 看到 assistant 流式响应
4. 看到工具时间线
5. 若出现危险命令，弹出审批卡片
6. 任务完成后会话自动进入历史

### 场景 3: 恢复历史会话

用户从左侧会话列表点开历史会话：

1. 加载消息历史
2. 加载时间线摘要
3. 可以继续追问
4. 后端继续复用原 `session_id`

### 场景 4: 诊断与调试

用户在任务失败时：

1. 查看时间线里失败的工具步骤
2. 打开 Logs 页面
3. 查看 sidecar 或 gateway 相关日志
4. 返回聊天继续修复

## 6. 范围

### 6.1 v1.0 In Scope

- Electron 桌面应用
- 本地 sidecar
- 现有 `web/` React 前端扩展
- Chat 页
- 会话侧栏
- 工具时间线
- 审批卡片
- 澄清卡片
- Setup Wizard
- Config / Sessions / Logs / Skills 复用
- Windows 与 macOS 安装包
- GitHub Actions 自动构建
- 自动更新

### 6.2 v1.0 Out of Scope

- Linux 发布
- 远程托管 agent
- 企业多用户协作
- 语音输入/输出作为主路径
- 把全部 messaging gateway 平台能力搬进桌面主流程
- 把 Hermes 所有可选 tools 默认全部打开

## 7. 功能需求

### FR-01 首启配置

系统必须提供首启向导，最少包含：

- provider 选择
- API key 输入
- 模型选择
- 默认工作区
- 审批模式

### FR-02 新建会话

系统必须支持创建桌面会话，并为其分配稳定 `session_id`。

### FR-03 按会话续跑

系统必须按 `session_id` 恢复消息历史与上下文，而不是要求前端回传完整对话。

### FR-04 流式聊天

系统必须支持 assistant 流式文本输出。

### FR-05 工具时间线

系统必须实时展示：

- 工具开始
- 工具结束
- 思考摘要
- 审批等待
- 澄清等待
- 运行完成/失败

### FR-06 审批流程

当工具操作存在风险时，系统必须暂停 run 并请求用户审批。

系统至少支持：

- 本次允许
- 当前会话允许
- 当前工作区允许
- 拒绝

### FR-07 澄清流程

当 agent 需要更多信息时，系统必须以卡片形式请求用户选择或输入答案。

### FR-08 会话管理

系统必须支持：

- 最近会话列表
- 会话搜索
- 删除会话
- 继续会话

### FR-09 设置与配置

系统必须允许用户查看和编辑 Hermes 配置。

### FR-10 日志与诊断

系统必须允许用户查看 sidecar 相关日志，帮助诊断任务失败或 runtime 问题。

### FR-11 技能管理

系统必须允许用户查看技能列表并启停技能。

### FR-12 更新

系统必须支持检查更新并提示安装。

## 8. 非功能需求

### NFR-01 启动性能

- 冷启动到主窗口出现：目标 < 5 秒
- 冷启动到 sidecar ready：目标 < 12 秒

### NFR-02 稳定性

- sidecar 崩溃后，Electron 应能自动重启并给出提示
- 重启后历史会话不能丢失

### NFR-03 本地安全

- sidecar 仅绑定 localhost
- 关键 API 需带 session token
- 危险命令必须显式审批

### NFR-04 可维护性

- 业务逻辑应集中在共享 Python 服务层
- Electron 只负责壳层与系统能力
- React UI 继续以 `web/` 为唯一前端源代码

### NFR-05 可发布性

- GitHub Actions 可以构建 Windows/macOS 产物
- tag 发布能自动上传 artifacts 到 GitHub Releases

## 9. 用户体验要求

### 9.1 开箱即用

用户绝不能被要求：

- 手动安装 Python
- 手动安装 Node
- 手动运行 `npm install`
- 手动运行 `agent-browser install`

### 9.2 可解释性

用户必须能看见 agent 关键动作，而不是只看最后一条回复。

### 9.3 可控性

用户必须能：

- 取消当前运行
- 拒绝危险操作
- 回答澄清问题
- 打开日志排查问题

## 10. 关键产品决策

### 决策 A: Chat 是首页

桌面版默认首页不是 `Status`，而是 `Chat`。

原因：

- 桌面产品的核心价值在于 agent 交互
- 配置、日志、技能属于管理面，不能成为主入口

### 决策 B: 时间线是主能力，不是附属面板

Hermes Desktop 的差异化不只是聊天，而是：

- 透明的 tool execution
- 可恢复的会话
- 审批与澄清闭环

所以时间线必须作为主视图的一部分。

### 决策 C: Setup Wizard 必须做减法

首启只展示能让用户开始工作的必要项。高级配置放到 Settings。

## 11. 页面级需求

### 11.1 Onboarding

页面必须支持：

- provider 选择
- key 输入
- 模型选择
- 工作区选择
- 审批模式选择
- 校验并完成

### 11.2 Chat

页面必须支持：

- 会话列表
- 新建会话
- 消息流
- 消息输入框
- 停止按钮
- 时间线
- 审批卡片
- 澄清卡片

### 11.3 Sessions

页面必须支持：

- 历史会话列表
- 搜索
- 删除
- 打开查看消息

### 11.4 Logs

页面必须支持：

- 日志文件切换
- 最近日志查看
- 失败时的错误上下文定位

### 11.5 Skills

页面必须支持：

- 技能列表
- 启用/禁用
- 技能描述查看

### 11.6 Config

页面必须支持：

- schema-driven 编辑
- 原始 YAML 编辑

## 12. 验收标准

### 12.1 安装验收

在全新机器上：

1. 安装包可以成功安装
2. 首次启动成功
3. 无需终端额外安装任何依赖

### 12.2 首启验收

1. 用户能通过向导完成 provider/API key/model/workspace 配置
2. 完成后直接进入聊天首页

### 12.3 聊天验收

1. 新建会话成功
2. 可看到流式消息
3. 可看到至少一种工具事件
4. 会话结束后在历史中可见

### 12.4 审批验收

1. 危险命令触发审批卡片
2. 用户批准后任务继续
3. 用户拒绝后任务被中止或改道

### 12.5 恢复验收

1. 重启应用后可重新打开历史会话
2. `session_id` 对应消息仍可用

### 12.6 打包验收

1. GitHub Actions 可自动构建 Windows/macOS artifacts
2. tag 发布时可上传 GitHub Release

## 13. 版本切分建议

### v1.0 必须完成

- 桌面壳
- sidecar
- Chat 首页
- 时间线
- 审批/澄清
- 会话恢复
- 打包与更新

### v1.1 可追加

- 更丰富的 workspace 管理
- 会话 fork
- 浏览器 runtime 分层下载
- 更强的 provider 管理

### v1.2 可追加

- 远程 host 模式
- 团队共享 profile
- 高级 agent telemetry

## 14. 风险

### 风险 1: 打包复杂度高于聊天功能本身

这是 v1.0 最大工程风险，但也是桌面产品价值所在。

### 风险 2: 审批/澄清桥接不完整会破坏体验

如果 run 不能可靠暂停与恢复，桌面版会退化成“只能看流式文本”的壳。

### 风险 3: 过度追求全量功能会拖慢发布

v1.0 必须围绕“coding-first、开箱即用、可解释”三点收敛。

## 15. 发布结论

`Hermes Desktop v1.0` 应被定义为一个产品化里程碑，而不是“额外前端”。

只要做到以下三件事，v1.0 就成立：

1. 用户安装即可运行
2. 用户能用首启配置进入完整 agent 聊天
3. 用户能在桌面中看清 agent 的动作、审批与结果

这三件事一旦闭环，Hermes 就从“强内核”变成“可交付产品”。
